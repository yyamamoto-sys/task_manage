# 管理者ガイド

管理者は、plan-app の**前提となる構造**を整える人です。Objective・KR・TF・メンバー登録、クォーターの TF 割り当てなど、業務メンバーが OKR サイクルに集中できるように下ごしらえします。

## 主な担当領域

| 領域 | 内容 | 頻度 |
|---|---|---|
| **Objective・KR** | 期初に登録、必要に応じて文言調整 | 期初＋随時 |
| **TF** | KR ごとに編成。クォーターまたぎで組替え | クォーター毎 |
| **TFのクォーター** | 設定 → タスクフォース のクォータータブ（1Q〜4Q）で、各TFがどのクォーターに属するかを直接切替 | クォーター毎 |
| **メンバー管理** | 参加者の追加・削除・短縮名 | 随時 |
| **PJ・タスク構造** | プロジェクト登録、ToDo・タスク階層の見直し | 随時 |

## 期初にやること

1. **Objective を登録**（設定 → Objective）
2. **KR を登録**（Objective に複数）
3. **メンバーを登録**（短縮名は2〜4字推奨）
4. **TF を登録**（1Q分でOK）
5. **クォーターの TF 割り当て**を確定
6. パイロットメンバーに URL とログイン手順を共有

→ 詳細： [Objective・KR・TF を登録する](../05_admin/objective-kr-tf.md)

## クォーター切替時にやること

1. KR代表に **クォーター計画**機能で次QのTF案を出してもらう
2. 案をレビューして TF を確定
3. 設定 → タスクフォース で**次Qのクォータータブ**に切り替え、継続するTFは「移動」、新設するTFは次Qのタブのまま新規作成
4. 必要に応じて TF メンバーを更新

TFのクォーターが未設定（前Qのまま等）だと、会議ノートで対象外の Q の TF まで表示されて混乱の元になります。**期初までに必ず次Qへの移動・新設を完了**してください。

## メンバー追加・削除

設定 → メンバー管理：
- 追加：短縮名・氏名・メアドを登録（ログイン用）。**メールアドレスは本人が実際にサインアップするアドレスと完全一致させること**（一致していないと自動的にはログインと紐づきません）
- 削除：論理削除（is_deleted）。履歴は残る
- 削除したメンバーは履歴に「（不明）」と表示されることがあるので、活躍したメンバーは無理に削除せず**短縮名末尾に "（退）" を付ける**運用がおすすめ

新しい部署自体を追加する手順、および「URLを送っただけの人が勝手にどこかの部署に紛れ込む」ことがない仕組みについては [新しい部署・メンバーを追加する](../05_admin/departments-and-members.md) を参照してください。

## AI使用量モニタ

設定 → AI使用量：
- intent ごとのトークン消費を確認
- 月次の傾向を見て、プロンプト改善やモデル切替を判断
- 突然の急増があれば原因調査（テストの繰り返し実行など）

## DBアクセス権限と機密

- Supabase の owner / service_role キーは**山本さんと管理者しか持たない**
- メンバーは Data API（supabase-js）経由で `authenticated` ロールでアクセス
- `members` / `projects` / `tasks` / `groups` / `task_dependencies` は部署（グループ）単位でRLS分離済み。他部署のデータは見えない（[新しい部署・メンバーを追加する](../05_admin/departments-and-members.md)参照）
- それ以外のOKR系テーブル（objectives/key_results/task_forces/todos等）は現状 `authenticated full access`（部署分離は未対応・今後対応予定）

## トラブル対応

| 症状 | 対応 |
|---|---|
| メンバーがログインできない | 設定 → メンバー管理で登録されているか確認 |
| エラー `PGRST205` | DBスキーマ未反映。山本さんに連絡 |
| エラー `42501` | DB権限不足。エラー本文の SQL を実行 |
| AI が応答しない | Anthropic API のキー切れ・レート制限を確認 |
| デプロイ後 真っ白 | Vercel デプロイ完了 + ブラウザリロード |
